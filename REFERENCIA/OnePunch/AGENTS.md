# AGENTS.md

Application name: OnePunch

Easy-to-use and customizable multi-agent system application.

It should be a simple interface where checking boxes activates an agent's functions and connections and modifies the message, and it should have a section for creating files that are saved as vector memory. For example, a user checks WhatsApp and call, and the agent will be able to interact with the user through these two channels, and if they also check the calendar and email boxes, the agent will be able to schedule meetings by validating the calendar to see available dates and times and also send emails. Certain checks can also define the tone of the target agent, for example, receptionist, and schedule pre-qualified calls. Each group of chefs must be at different levels according to the category of checks. When certain chefs are activated, the multi-agent system, tools, and prompts are activated at different levels of the system. 

Checks must have all available options, channels, tools, tones, objectives, etc., to have all available options.

They must be prepared to meet 360% of the objectives for prospects and customers, and for different business models, i.e., you can both schedule calls and sell via messaging, and have access to inventory via Google Drive sheets where we can see inventory, prices, codes, and so on, and use this for transactions, and charge with PayPal but in a conversational way, either by text or messaging on different messaging channels or by call, as well as scheduling calls.

When certain chefs are checked, the integration fields and methods must be enabled. Always look for the easiest ways to connect, such as logins, QR codes, or forms to enter connection data for different levels and functionalities. 

The different agents in the multi-agent system will have context from the RAG data. This, together with the prompts and connections, will define the overall context of the agent. 

The interface should prioritize simplicity and minimalism, but be modern, dynamic, easy to use, and responsive for use on different screens in web browsers. The interface will be used primarily for the configuration and analysis of users, calendars, funnels, and graphs with easy-to-understand summaries and a chat window for the administrator to interact with their data and company information, conversation reports, opportunities, tasks, etc. Communication will work on two levels: on the one hand, customers and, on the other, the company. It can be with one or more company contacts depending on the levels, through calls and messages, using specific numbers that the agent will understand, along with the role and name of the user, and will have a context of the conversations with these administrator users, through messages or calls. In this way, the multi-agent system interacts with potential customers and meets configuration requirements, while sharing data in real time, responding to queries, and providing information when requested by the administrator or company employees. It can also communicate with one or another employee depending on the specific need for information or the special requirements of users, or respond to questions and lack of context or settings, if they are not found in the documentation or memory, etc. Based on this, it can assist users, answer their questions, and much more, thus functioning as a real department of a company. or settings, if they are not found in the documentation or memory, etc., and based on this, help users, answer their questions, etc. In this way, it can function as a real department of a company, specifically the customer service and company interaction department. In the configuration panel for each company/business, the interface administrator can define which model to use and add the corresponding API keys. It must have as many standard LLM models as possible, as well as voice models, in order to function. In this way, the platform will adapt to work with the business's own consumption patterns, adding specific classes and activating the models to be used. In this way, the platform offers cloud-based software with specific spaces. The system will use Python with Flask, and for vector memory, we can use Pinecone for rag and MySQL for administrator users and companies, linking it to each user's rag index if necessary. Each panel per company can link to different employees and their phone numbers, emails, or WhatsApp numbers, and each company will have a field to enter the agent's name, which is actually an agent system, but this humanizes the personalization. In addition, the user will be able to select the gender of the agent, whether female or male, and will also be able to establish a personality or specific instructions that give that touch of originality and hyper-personalization to the traits or ways of expressing themselves, which will be the prompt of the system that integrates and combines with the different prompts and other features that adapt to the chefs' panel, all of which defines the identity and personality of the agent (multi-agent system).

The system will run in a /venv/ and the key credentials in the .env.
